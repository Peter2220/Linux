# rsyslog - Centralized Logging Summary

## Purpose

rsyslog (Rocket-fast System for Log Processing) handles local system logging, message filtering, routing, and centralized network-based log collection.

---

## Architecture & Data Flow

rsyslog processes data using a modular pipeline. Logs enter through **Inputs**, pass through the core engine's **Filters & Rulesets**, and are delivered via **Outputs**.

* **Client (Forwarder):** Generates local logs and forwards them across the network.
* **Server (Receiver):** Listens on the network, filters logs, and stores them systematically.

---

## Syslog Filtering Concepts (Facilities & Severities)

Rsyslog rules rely on a `facility.priority` selector logic. The **Facility** represents the subsystem that generated the log, while the **Severity (Priority)** defines its urgency.

### Syslog Facilities Reference

| Facility Code | Facility Name | Description |
| --- | --- | --- |
| `auth` / `authpriv` | Authentication | Security/authentication messages (logins, sudo usage) |
| `cron` | Clock Daemon | Scheduled background jobs (cron, anacron) |
| `daemon` | System Daemons | Background system services without explicit facilities |
| `kern` | Kernel | Core operating system kernel messages |
| `mail` | Mail System | Electronic mail servers (Postfix, Sendmail) |
| `syslog` | Internal Syslog | Messages generated internally by rsyslog itself |
| `user` | User-level | Generic user-space applications (default for `logger`) |
| `local0` – `local7` | Custom Local | Reserved for administrators to map custom applications |

### Syslog Severities Reference (Highest to Lowest)

| Numerical Code | Severity Level | Description |
| --- | --- | --- |
| 0 | `emerg` | System is unusable (Emergency) |
| 1 | `alert` | Action must be taken immediately |
| 2 | `crit` | Critical conditions |
| 3 | `err` | Error conditions |
| 4 | `warning` | Warning conditions |
| 5 | `notice` | Normal but significant conditions |
| 6 | `info` | Informational messages |
| 7 | `debug` | Debug-level messages |

## Understanding rsyslog Modules

Modules provide the core capabilities for each stage of the logging pipeline. They use a standard naming convention based on their function.

### 1. Input Modules (Prefix: `im`)

Input modules gather log streams from various sources and ingest them into the rsyslog engine.

* **`imudp`:** Opens a UDP network socket (usually port 514). It provides rapid log ingestion without connection overhead, though it does not guarantee packet delivery.
* **`imtcp`:** Opens a reliable, connection-oriented TCP network socket, ensuring logs are not dropped in transit.
* **`imuxsock`:** Ingests local logs generated by system processes and standard command-line tools (like `logger`).
* **`imjournal`:** Imports binary logs directly from the modern `systemd-journald` process.

### 2. Output Modules (Prefix: `om`)

Output modules format and write logs out to their final destinations once processing is complete.

* **`omfile`:** Writes logs to physical files on disk (e.g., `/var/log/messages`). It supports dynamic file generation when combined with templates.
* **`omfwd`:** Transmits logs over the network to a centralized remote collector.
* **`omprog`:** Pipes log messages into custom scripts or binaries to trigger automated alerts or third-party workflows.

## Client Configuration

To forward logs to a centralized server, place configurations inside `/etc/rsyslog.d/forward.conf`.

```bash
# Forward all logs (*.*) to a remote server over TCP on port 514
*.* action(type="omfwd" target="192.168.1.50" port="514" protocol="tcp")

# Example: Forward ONLY authentication/security logs over UDP
authpriv.* action(type="omfwd" target="192.168.1.50" port="514" protocol="udp")

```

### Testing Log Generation

Use the `logger` utility to inject test messages directly into the local logging facility to verify routing.

```bash
# Generate a standard test message
logger -p user.info "Test message from client node"

# Generate an authentication alert to test secure log routing
logger -p authpriv.alert "Unauthorized login attempt detected"

```

## Server Configuration

To allow a machine to act as a centralized receiver, edit `/etc/rsyslog.conf` to load network modules and define listening ports.

```bash
# Enable UDP Reception
module(load="imudp")
input(type="imudp" port="514")

# Enable TCP Reception
module(load="imtcp")
input(type="imtcp" port="514")

```

### Dynamic Log Storage (Templates)

Instead of dumping all incoming client logs into one massive file, use rulesets and templates to organize storage folders dynamically by source hostname.

```bash
# Define a file path template based on the sender's hostname
template(name="PerHostLogs" type="list") {
    constant(value="/var/log/syslogtest/")
    property(name="hostname")
    constant(value="/syslog.log")
}

# Apply the template to save all incoming remote logs
*.* action(type="omfile" DynaFile="PerHostLogs")

```

**Expected Directory Structure Result:**

```text
/var/log/syslogtest/bastion/syslog.log
/var/log/syslogtest/web-server01/syslog.log

```

## Additional Advanced Configuration Examples

### Example 1: Discarding/Dropping Unwanted Logs (Spam Control)

Stop specific chatty services from filling up storage disks.

```bash
# Drop all cron logs completely before they write anywhere else
if $programname == 'CROND' then stop

```

### Example 2: Routing Messages to Custom Local Files

Filter logs based on their priority or system component origins.

```bash
# Store all kernel logs in a separate dedicated file
kern.* /var/log/kernel_errors.log

# Store emergency messages only across everything
*.emerg                                                /var/log/critical_emergencies.log

```

## Key Insights & Troubleshooting

### Diagnostic Status Check

Whenever configuration properties are updated, the service daemon must be validated and restarted.

```bash
# Allow port 514 through Firewall
firewall-cmd --add-port=514/tcp --permanent && firewall-cmd --add-port=514/udp --permanent

# Check configuration syntax before restarting
rsyslogd -N 1

# Restart service to apply new filters
systemctl restart rsyslog
```

### Managing Connection Drops

As observed in the log files (`action 'omfwd' suspended`), temporary drops can occur due to:

* Firewall timeouts or strict network Access Control Lists (ACLs).
* Central log receiver service restarts.

Note: rsyslog natively handles these states safely. It buffers events in an internal queue and automatically retries connections until the channel is successfully resumed.
